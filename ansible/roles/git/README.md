# roles/git

Deploys git configuration and manages multi-context project identities.

## Design

### File ownership

| File | Managed by | Force on re-run |
|------|-----------|----------------|
| `~/.gitconfig` | Ansible | Yes — stable template, safe to re-render |
| `~/.config/git/project-includes` | Ansible + shell functions | Yes — fully generated |
| `~/.config/git/projects.yml` | Ansible (first run) then shell functions | No — live source of truth |
| `~/.config/git/profiles/<n>.inc` | Ansible (first run) then shell functions | No — user-editable after creation |
| `~/.config/git/profiles/local.inc` | User only | No — never touched after creation |
| `~/.config/git/ignore` | Ansible | Yes |
| `~/.config/git/attributes` | Ansible | Yes |

### How `~/.gitconfig` is managed

`~/.gitconfig` is a **generated artefact**, not a source file. It is rendered from
`templates/gitconfig.j2` plus `host_vars/localhost.yml` on every Ansible run.
It is **not** synced back to the repo by the sync service — the template is the source.
Do not add content to `~/.gitconfig` directly; use `~/.config/git/profiles/local.inc`
for machine-local overrides.

### How project identities work

```
~/.gitconfig
    └── [include] → ~/.config/git/project-includes
                        └── [includeIf "gitdir:~/Projects/Personal/GitHub/"]
                                → ~/.config/git/profiles/personal-github.inc
                                    [user]
                                        email = you@personal.com
                                        signingkey = FINGERPRINT
```

`project-includes` is rendered by Ansible from `host_vars` on every run, and also
regenerated by the shell functions after any add/remove operation. The manifest
(`projects.yml`) and `host_vars/localhost.yml` are kept in sync by the shell functions,
so Ansible re-runs remain safe.

### Ansible ↔ shell function contract

When `git-add-project` / `git-update-project` / `git-remove-project` run:

1. They update `~/.config/git/projects.yml` (the live manifest)
2. They update `~/.config/git/project-includes` (the includeIf file)
3. They create or edit `~/.config/git/profiles/<n>.inc` via `git config --file`
4. **If `DOTFILES_REPO_DIR` is set**, they update `host_vars/localhost.yml`

If `DOTFILES_REPO_DIR` is not set (e.g. first shell session after a manual clone),
the functions warn and continue. The manifest and includes are still updated correctly
for the local machine, but `host_vars` will be stale. The next Ansible run will
re-render `project-includes` from the stale `host_vars`, overwriting local additions.

**Set `DOTFILES_REPO_DIR` in `env/00-core.sh`** to avoid this:

```sh
export DOTFILES_REPO_DIR="${HOME}/Projects/Personal/GitHub/dotfiles"
```

If `host_vars` and the manifest have diverged, recover with:

```sh
git-sync-projects --status          # review differences
git-sync-projects --from-host-vars  # rebuild manifest from host_vars
# or: delete projects.yml and re-run Ansible to reset from host_vars
```

## Variables

All variables are defined in `group_vars/all.yml` with empty defaults.
Personal values belong in `host_vars/localhost.yml` (gitignored).

| Variable | Default | Description |
|----------|---------|-------------|
| `projects_base` | `~/Projects` | Root directory for all project directories |
| `git_name` | `""` | Global display name |
| `git_default_email` | `""` | Global email (fallback when no project matches) |
| `git_default_signing_key` | `""` | Global GPG fingerprint (enables global gpgsign) |
| `git_editor` | `vim` | Git editor |
| `git_diff_tool` | `vimdiff` | Diff tool |
| `git_merge_tool` | `vimdiff` | Merge tool |
| `git_projects` | `[]` | List of project definitions (see below) |

### `git_projects` entry fields

| Field | Required | Description |
|-------|----------|-------------|
| `context` | Yes | Life/work area — `Personal`, `Acme`, `ClientName`, etc. |
| `provider` | Yes | Git host — `GitHub`, `GitLab`, `AzureDevOps`, etc. |
| `email` | Yes | Identity email for repos under this context/provider |
| `signing_key` | No | GPG fingerprint — enables `gpgsign` for this context |
| `name` | No | Display name override — defaults to `git_name` |
| `ssh_key` | No | SSH key path — informational, consumed by the `ssh` role |

## Shell functions

Sourced via `shell/config/tools/git.sh` (guarded by `command -v git`).
Requires `yq` v4 (mikefarah/yq) for manifest operations.

### Project management

```sh
git-add-project <context> <provider> <email> [signing-key] [name]
    Add a new context/provider pair. Creates the directory, profile .inc,
    and updates the manifest, project-includes, and host_vars.

git-update-project <context> <provider> [--email <e>] [--signing-key <k>] [--name <n>]
    Update fields on an existing project. Updates the .inc file, manifest,
    and host_vars in sync. Multiple flags can be combined.

git-remove-project <context> <provider>
    Remove a project from config. Prompts for confirmation. Does NOT delete
    the project directory or any repos inside it.

git-list-projects
    Tabular view of all configured projects from the manifest.

git-sync-projects
    Ensure all manifest entries have their directory and profile .inc file.
    Rebuilds project-includes. Safe to re-run.

git-sync-projects --status
    Show which projects are missing their directory or profile file.
    No changes made.

git-sync-projects --from-host-vars
    Rebuild the manifest from host_vars/localhost.yml. Use after Ansible
    changes that weren't propagated to the live manifest.

git-projects-base
    Print the resolved projects base directory.
```

### Examples

```sh
# Add a new personal GitLab context
git-add-project Personal GitLab me@personal.com

# Add a new employer with a signing key
git-add-project Acme AzureDevOps me@acme.com GPGFINGERPRINT "Graham Watts"

# Update signing key after key rotation
git-update-project Personal GitHub --signing-key NEWFINGERPRINT

# Change email on a work context
git-update-project Acme GitHub --email newrole@acme.com

# Check what's configured
git-list-projects

# Check sync status without making changes
git-sync-projects --status
```
