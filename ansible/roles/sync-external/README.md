# sync-external role

Generic external add-on repo sync engine. Replaces the old per-repo `nvim`
and `ai-tools` roles with one engine that clones/adopts any number of add-on
repos listed in `external_synced_repos`, keeps them pulled on an hourly
timer, and deploys files from each according to that repo's own
`.dotfiles-sync.yml` manifest (see
[docs/sync-manifest-spec.md](../../../docs/sync-manifest-spec.md)).

The dotfiles repo owns the engine; each add-on repo owns the description of
what to do with itself. This role never hardcodes another repo's internal
layout.

## Activation

Runs for the `workstation` and `server` profiles, gated on
`dotfiles_sync_enabled | bool` (the same flag that gates the `sync` role).
It is not profile-restricted the way the old `nvim`/`ai-tools` roles were —
generic add-on sync is useful on servers too.

With `external_synced_repos: []` (the default), the role still deploys the
timer/service units but the per-repo loop is a no-op — there is nothing to
sync until at least one repo is registered.

## What this role does

Once for the whole run, before any repo is processed:

- Resolves the `dest` denylist (see [dest validation](#dest-validation)
  below) to absolute paths and a search pattern — computed once since it
  only depends on `$HOME` and the role's own defaults, not on any repo.

Then, per entry in `external_synced_repos` (see
[docs/external-sync.md](../../../docs/external-sync.md) for the full field
reference):

1. Resolves `clone_dir` to an absolute path.
2. Ensures the repo's config/state directories exist.
3. **Normalises the repo URL** — public/private aware (see below).
4. **Adopts or clones** the repo (see below).
5. Reads `.dotfiles-sync.yml` from the clone, if present, and:
   - validates every `deploy[].src` stays within `clone_dir`;
   - validates every `deploy[].dest`/`dest_macos` resolves under `$HOME` and
     isn't on the [dest denylist](#dest-validation);
   - validates `hooks.post_deploy.command[0]`, if the manifest declares a
     hook, the same way `src` is validated.
6. Renders `sync.conf` (created once, never overwritten by Ansible).
7. Renders `deploy.list` from the parsed manifest (regenerated every run).
8. Determines the `allow_hooks` gate state (manifest declares a hook AND
   `repo.allow_hooks` is `true`) and prints a one-line reminder — without
   failing the run — when a hook is declared but not allowed.
9. Computes the manifest's `git hash-object` and writes it to
   `manifest-hash` — the single source of truth `scripts/external-sync.sh`
   compares against to detect and warn on manifest drift (see [Manifest
   drift detection](#manifest-drift-detection) below).
10. Renders `hooks.list` from the parsed manifest — empty when no hook is
    declared or the `allow_hooks` gate is closed.
11. Runs `external-sync <name>` once for an initial deploy (and, if
    applicable, the repo's first hook run).

Then, once for the whole run (not per repo):

- Deploys `scripts/external-sync.sh` as `~/.local/bin/external-sync`.
- Deploys and enables the `external-sync.timer` (Linux) or
  `com.external-sync.plist` (macOS) — hourly cadence, single unit for all
  registered repos.

## Public vs private repos

| | `private: false` | `private: true` |
| --- | --- | --- |
| Transport | HTTPS, as given in `repo_url` | SSH, rewritten to an alias — works for any git host |
| Deploy key | None | `~/.ssh/dotfiles-<name>`, alias `dotfiles-<name>` |
| URL rewrite | None — used verbatim | `<any form>://<host>/owner/repo.git` (or `git@host:owner/repo.git`) → `git@dotfiles-<name>:owner/repo.git`; the owner/repo path is extracted regardless of host, so re-running against an already-rewritten URL is a no-op |

Deploy keys and SSH aliases are generated by `install.sh` (see
[Phase 5](../../../docs/external-sync.md#adding-a-repo)), not by this role —
the role only performs the URL rewrite so cloning uses the right alias.

## dest validation

`deploy[].src` (and `hooks.post_deploy.command[0]`) may only be a path
relative to the repo root, no leading `/`, no `..` segments — the manifest
can only ever make the engine *read* from within the clone. `deploy[].dest`
(and `dest_macos`) is the *write* direction and is held to a stricter rule:
it must resolve under `$HOME`, contain no `..` segments, and must not target
a denylist of paths this role has no business writing into — see
`defaults/main.yml`'s `sync_external_dest_denylist_dirs`/
`sync_external_dest_denylist_files` for the exact list, and
[docs/sync-manifest-spec.md](../../../docs/sync-manifest-spec.md#dest-validation)
for the rationale. A manifest violating any of these rules fails the
Ansible run, naming the repo and the offending path.

## Hooks

A manifest may declare a single `hooks.post_deploy` (schema v1's only
event). Two gates must both be open before it ever runs — see
[docs/sync-manifest-spec.md's hook contract](../../../docs/sync-manifest-spec.md#hook-contract)
for the full contract (schema, `run_on` semantics, environment variables,
failure handling):

1. The manifest declares it.
2. This repo's `external_synced_repos` entry sets `allow_hooks: true`
   (default `false`).

When (1) is true and (2) is false, `hooks.list` is rendered **empty** (never
omitted) and a `ansible.builtin.debug` names the repo and the exact
remediation — the run still succeeds. This role never fails a provisioning
run just because an add-on repo added a hook a machine hasn't opted into.

## Manifest drift detection

`deploy.list`/`hooks.list` are rendered once per Ansible run from whatever
the manifest looked like at that moment — the `external-sync` timer only
ever re-reads those already-rendered files, it never re-parses
`.dotfiles-sync.yml` itself (Bash never parses YAML — see the spec's
[Purpose](../../../docs/sync-manifest-spec.md#purpose) section). This role
writes `manifest-hash` (`git hash-object` of the manifest, or the literal
string `absent` when there is none) so `scripts/external-sync.sh` can detect
when the manifest in the clone has moved on from what was last rendered, and
warn — in the repo's log and in `external-sync --status` — rather than
silently doing nothing with an upstream manifest change. Only this role
writes `manifest-hash`; the script only ever reads it.

## Adopt vs clone

`ansible.builtin.git` is called with `update: false`, which makes the
following naturally idempotent:

- `clone_dir` doesn't exist → cloned fresh.
- `clone_dir` exists and is a valid git repo → **adopted in place**: no
  re-clone, no pull. Ongoing pulls are the `external-sync` timer's job, not
  Ansible's — this role only ever touches a repo's working tree on first
  contact.
- `clone_dir` exists and is **not** a git repo (e.g. a config directory that
  predates this system) → backed up to `<clone_dir>.bak-YYYY-MM-DD`, then
  cloned fresh. Nothing is deleted.

## Variables

| Variable | Default | Override in | Purpose |
| --- | --- | --- | --- |
| `external_synced_repos` | `[]` | `host_vars/localhost.yml` | List of repos to sync. See field reference below. |
| `external_sync_conf_dir` | `~/.config/external-sync` | `group_vars/all.yml` | Base directory for per-repo `sync.conf`/`deploy.list`. |
| `external_sync_state_dir` | `~/.local/share/external-sync` | `group_vars/all.yml` | Base directory for per-repo `last-sync`/logs. |

### `external_synced_repos` entry fields

| Field | Required | Purpose |
| --- | --- | --- |
| `name` | yes | Unique identifier — used for the config/state directory, the SSH alias (private repos), and as the sync-script argument. |
| `repo_url` | yes | HTTPS URL (public) or HTTPS/SSH/alias URL (private — rewritten to the alias form automatically). |
| `clone_dir` | yes | Where the repo is cloned. May use `~`. |
| `private` | yes | `true`/`false` — controls the URL rewrite and whether a deploy key is expected. |
| `allow_hooks` | no | `true`/`false`, default `false` — whether this repo's declared `hooks.post_deploy` (if any) is allowed to run. See [Hooks](#hooks). |

Cadence (hourly) and deploy rules are **not** configured here — cadence is
the engine's fixed decision, deploy rules come from the repo's own
`.dotfiles-sync.yml`.

## On-disk layout

```
~/.config/external-sync/<name>/sync.conf     # REPO_URL, CLONE_DIR, GIT_BRANCH, DEV_MODE
~/.config/external-sync/<name>/deploy.list   # src|dest|mode|force|executable (empty = clone-only)
~/.config/external-sync/<name>/hooks.list    # event|run_on|timeout|argv... (empty = no hooks / gated off)
~/.local/share/external-sync/<name>/last-sync
~/.local/share/external-sync/<name>/manifest-hash     # written by this role only — see Manifest drift detection
~/.local/share/external-sync/<name>/hook-ran          # run_on: initial sentinel — written by the script only
~/.local/share/external-sync/<name>/last-hook-status  # <ISO8601> <ok|failed> <rc> <event>
~/.local/share/external-sync/<name>/logs/sync.log
~/.local/bin/external-sync                   # symlink → scripts/external-sync.sh
~/.config/systemd/user/external-sync.service # Linux only
~/.config/systemd/user/external-sync.timer   # Linux only
~/Library/LaunchAgents/com.external-sync.plist  # macOS only
```

The running timer discovers repos from `~/.config/external-sync/*/` at
runtime, not from `host_vars` — this decouples the timer from Ansible.

## Idempotency notes

- `sync.conf` is written with `force: false` — never overwritten by Ansible,
  preserving runtime changes to `DEV_MODE` or `GIT_BRANCH`. Only its
  `REPO_URL` line is kept current via `lineinfile`.
- `deploy.list`/`hooks.list` are pure derived data and are regenerated every
  **Ansible** run — not every sync cycle. A manifest change upstream needs
  an Ansible re-run to take effect; see [Manifest drift
  detection](#manifest-drift-detection).
- The timer/service `enabled`/`loaded` state is checked and (re-)applied via
  an unconditional standalone task on every run, not a handler — so it is
  never accidentally left disabled just because nothing changed that run.
